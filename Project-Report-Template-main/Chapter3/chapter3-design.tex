\chapter{Analysis and Design}

This chapter explains the internal design of VisuaLearn AI. The application has to support ordinary chat, focused learning mode, document-assisted answers, generated study material, simulations, visualizations, feedback, and saved learning history. To keep these features manageable, the design separates the browser interface, API routing, learner-context handling, AI agents, validation, storage, and feedback processing.

\section{Requirement Analysis}

The main functional requirements fixed during design were:
\begin{enumerate}[leftmargin=1.2cm]
\item Users must be able to register, log in, and access protected learning features.
\item The application must provide a chat interface for educational questions and streamed AI responses.
\item Conversations, messages, learning resources, and generated artifacts must be stored for later use.
\item The system must generate explanations, quizzes, flashcards, mind maps, simulations, image outputs, video outputs, and 3D visualizations when the selected learning task requires them.
\item Agent-based modules must handle context building, strategy selection, content generation, validation, and feedback handling.
\item Uploaded documents must be available for document-based learning through retrieval over stored resources.
\item Feedback and interaction signals must be recorded so that later responses can use learner history.
\item Voice input and speech output must be supported when the required services are configured.
\end{enumerate}

The non-functional requirements were:
\begin{enumerate}[leftmargin=1.2cm]
\item \textbf{Scalability:} New learning features should be added through separate routes, services, and UI modules instead of changing one large component.
\item \textbf{Reliability:} Failed AI responses, visualization errors, or unavailable provider services should return a controlled fallback rather than leaving the learner with a blank screen.
\item \textbf{Performance:} Streaming should reduce waiting time for chat responses, and expensive generation routes should be rate limited.
\item \textbf{Security:} User records, uploaded files, and provider keys should be protected through authentication, ownership checks, backend-only secrets, and safe rendering.
\item \textbf{Maintainability:} Frontend components, backend routes, services, and agents should have clear responsibilities so that faults can be isolated during testing.
\end{enumerate}

The frontend uses React, Vite, and Tailwind CSS because the interface contains many reusable views, frequent state changes, and responsive layouts \cite{ReactDocs,ViteDocs,TailwindDocs}. Azure OpenAI services are used only for AI-provider features that are configured in the deployment environment \cite{SupabaseDocs,PostgreSQLDocs,AzureOpenAIDocs}.

Figure \ref{fig:architecture} shows the main layers of the system. The presentation layer contains the chat screen, learning workspace, and artifact viewers. The application layer receives requests, applies route handling, and manages session information. The intelligence layer builds learner context and selects a learning strategy. \section{Adaptive Strategy Selection Workflow}

The strategy selector runs after context building and before content generation. It uses the current query, conversation history, learner profile, retrieved documents, and feedback history to decide the next response form. The selected response form may be a concise answer, guided explanation, quiz, flashcard set, simulation, visual output, remediation, or video generation request.

\begin{center}
\texttt{User Query -> Context Builder -> Strategy Selector -> Content Generator -> Simulation/Visualization -> Validation -> Storage -> Feedback}
\end{center}

\begin{tcblisting}{listing only,colback=gray!10!white, breakable, boxsep=0pt,top=1mm,bottom=1mm,left=1mm,right=1mm,
listing options={basicstyle=\footnotesize\ttfamily,backgroundcolor=\color{gray!20},breaklines=true,columns=fullflexible}}
Input: user query, conversation history, profile, document chunks, feedback
1. Build context from current query, profile, history, and retrieved documents.
2. Classify intent as normal chat, deep learning, quiz, flashcard, simulation, visualization, video, or voice.
3. Estimate learner state using confidence, engagement, failures, topic status, and response time.
4. Select strategy: concise explanation, guided steps, visual explanation, quiz, simulation, remediation, or generated video.
5. Generate the required learning artifact using the selected agent.
6. Validate output for safety, structure, and rendering support.
7. Store message, resource, selected strategy, and metadata.
8. Collect feedback and update future strategy context.
\end{tcblisting}

\section{Security Design}

The security design covers user access, ownership, provider keys, cost control, and generated output.
\begin{itemize}[leftmargin=1.2cm]
\item \textbf{Authentication:} Supabase sessions identify signed-in users before private routes are accessed.
\item \textbf{Authorization:} Ownership checks ensure that users can access only their own conversations, resources, documents, and feedback.
\item \textbf{Rate limiting:} Expensive routes such as chat, simulation, visualization, voice, and generation APIs are limited to control misuse and cost.
\item \textbf{Safe rendering:} Generated visual artifacts are validated or rendered in controlled viewers before display.
\item \textbf{Backend-only secrets:} Provider keys are stored on the backend and are never exposed to the browser.
\end{itemize}

\section{Design Rationale}

React was chosen because the application has many stateful views, including chat messages, learning tabs, artifact cards, feedback controls, and viewers. Vite was used because it keeps frontend development builds fast and the configuration small. Node.js and Express were used on the backend because the project needs lightweight routing, middleware, streaming, and service orchestration in the same JavaScript ecosystem.

Supabase was chosen because it provides authentication, PostgreSQL-backed storage, file storage, and ownership-oriented access control in one service. Server-Sent Events were used for chat streaming because they allow the browser to receive partial responses over an ordinary HTTP connection. This is useful in a learning interface because the learner can start reading while the answer is still being produced.

The agent-based design was used because each learning task has a different output structure. A quiz generator, flashcard generator, document retrieval module, simulation planner, video generator, validation service, and feedback handler cannot all be treated as the same type of response. Separate agents keep these responsibilities visible and easier to test.

\section{Design Summary}

The Chapter 3 design keeps the learning workflow traceable from the user request to the saved result. A request passes through context building, strategy selection, content generation, validation, storage, and feedback. This structure allows VisuaLearn AI to support both normal conversation and focused learning mode while still keeping generated artifacts, learner records, and feedback organized.
